## 3. ZK Proof Pipeline

### 3.1 Off-Chain: Proof Generation

| Stage | Time (estimated) | Notes |
|-------|------------------|-------|
| **Witness generation** | 1–3 seconds | Noir JS runtime |
| **Proof generation** | 5–15 seconds | UltraHonk via Barretenberg WASM |
| **Total off-chain** | **6–18 seconds** | Single-threaded; parallelizable |

The proof is generated client-side (browser or server) using `@aztec/bb.js`. No gas cost — this is pure computation.

### 3.2 On-Chain: Proof Verification

The `HonkVerifier.verify()` contract uses heavy EVM precompile operations:

| Precompile | Address | Gas Cost Per Call | Calls in Verify |
|------------|---------|-------------------|-----------------|
| `ecAdd` (BN254) | `0x06` | 150 gas | ~50–100 |
| `ecMul` (BN254) | `0x07` | 6,000 gas | ~30–50 |
| `modexp` | `0x05` | variable | ~10–20 |

**Estimated HonkVerifier.verify() gas: 350,000–500,000**

This estimate is based on:
- Circuit size N=65,536 (log N=16 rounds)
- 19 public inputs (16 Honk internal + 3 user)
- UltraHonk pairing-based verification
- Industry benchmarks for similar Honk verifiers

> **Why we can't measure exactly in unit tests:** The `HonkVerifier` requires a valid proof generated by the Noir circuit. Unit tests use `MockLoanProofVerifier` for speed. The real verifier gas can only be measured via integration test with FFI proof generation (`TestVerifierOnly.s.sol`).

### 3.3 On-Chain: Poseidon2 Hash Recomputation

After proof verification, `createLoan()` recomputes the loan hash on-chain:

```
Poseidon2.hash([borrowerCommitment, uwKeyX, uwKeyY, tierId, 
                principal, apr, origFee, term, industry, 
                timestamp, loanId])  // 11 field elements
```

| Implementation | Gas (11 inputs) | Notes |
|----------------|-----------------|-------|
| **MockPoseidon2** (keccak fallback) | 2,629 | Used in unit tests |
| **Poseidon2Yul** (real) | ~50,000–80,000* | Inline Yul, constant-time |
| **Poseidon2 Huff** (real) | ~30,000–50,000* | Huff assembly, most efficient |

*\* Estimated; requires integration test with real Poseidon2 deployment.*

## 6. ZK Circuit Details

### Circuit Architecture

| Property | Value |
|----------|-------|

| **ACIR Opcodes** | **671** |

| **Backend Gate Count** | **6,248** (actual constraint count) |
| **Padded Circuit Size (N)** | 8,192 (2¹³, next power-of-two for SRS) |
| **Expression Width** | Bounded { width: 4 } (UltraPlonk fan-in) |
| **Source lines** | 3,402 (608 circuit logic + 2,794 inline tests) |
| **Compiled artifact** | 175 KB (`circuits.json`) |
| **Verification key** | 3,680 bytes |
| **Noir tests** | **78 tests, 78 passed** |

> ⚠️ **Correction from earlier estimates:** Previous references to "N=65,536 (2¹⁶)" were based on the Solidity verifier's hardcoded SRS domain. The actual backend gate count measured via `bb gates` is **6,248**, which pads to N=8,192 (2¹³). The HonkVerifier VK was generated for a larger domain — this is safe (over-padded) but means on-chain verification is slightly more expensive than strictly necessary.

### Constraint Breakdown by Circuit Function

Measured: 671 ACIR opcodes → 6,248 backend gates (after black-box expansion).

| Circuit Step | Function | Constraint-Heavy Operations | Est. Gates | % of Total |
|-------------|----------|---------------------------|-----------|------------|
| **0. Loan hash** | `compute_loan_hash()` | Poseidon2 hash (11 inputs) | ~800 | 13% |
| **1. Borrower commitment** | `verify_borrower_commitment()` | Poseidon2 hash (5 inputs) | ~500 | 8% |
| **1a. Nullifier** | inline `Poseidon2::hash()` | Poseidon2 hash (4 inputs) | ~450 | 7% |
| **1b. Attestation freshness** | `verify_attestation_age()` | 4 u64 comparisons + multiply | ~80 | 1% |
| **1c. Schnorr signature** | `verify_schnorr_signature()` | `multi_scalar_mul` (2-MSM) + Poseidon2(3) + byte decomp | **~2,500** | **40%** |
| **2. Policy hash** | `compute_policy_hash()` | Poseidon2 hash (21 inputs) | ~1,200 | 19% |
| **3. Policy compliance** | `validate_policy_compliance()` | 15+ u64 comparisons + u64 multiply + asserts | ~350 | 6% |
| **Overhead** | Input marshalling, range checks, wiring | Implicit Noir overhead | ~370 | 6% |
| **Total** | | | **6,248** | 100% |

### Key Observations

1. **Schnorr signature verification dominates (40%):** The `multi_scalar_mul` blackbox opcode (R' = s·G + e·PK over Grumpkin) is the single most expensive operation. This is a 2-point multi-scalar multiplication over BN254's embedded curve — it's implemented as a native Barretenberg opcode, which is ~10× cheaper than ECDSA secp256k1 would be, but still the circuit's bottleneck.

2. **Poseidon2 hashing is the second-largest cost (47% combined):** Five Poseidon2 calls with varying input sizes:
   - `compute_policy_hash()` — 21 inputs (~1,200 gates) — **largest single hash**
   - `compute_loan_hash()` — 11 inputs (~800 gates)
   - `verify_borrower_commitment()` — 5 inputs (~500 gates)
   - Nullifier hash — 4 inputs (~450 gates)
   - Schnorr challenge hash — 3 inputs (~350 gates, included in Schnorr total)
   
   Poseidon2 uses custom gates in UltraHonk, making it ~5× cheaper than if implemented with standard arithmetic constraints.

3. **Policy compliance checks are cheap (6%):** The 15+ inequality comparisons (`assert(a >= b)`, `assert(a <= b)`) and the loan-to-EBITDA cross-multiplication are lightweight — each u64 comparison is ~15–20 gates.

4. **Circuit is compact:** 6,248 gates is small by ZK standards. For comparison:
   - Tornado Cash (Groth16): ~30,000 constraints
   - Semaphore v4: ~16,384 constraints
   - zkEmail: ~1,000,000+ constraints
   - This means fast proof generation and cheap verification.

- **Computation gas** is ~10–50x cheaper than L1
- **Data publication** (L1 calldata) is the dominant cost

## 8. Comparison with Industry Benchmarks

### ZK Proof Verification Gas

| Protocol / System | Proof System | Circuit Size | Verify Gas | Notes |
|-------------------|-------------|--------------|------------|-------|
| **Credit Rail** | UltraHonk | **6,248** | ~350,000–500,000* | Aztec BB, BN254 |
| Tornado Cash | Groth16 | ~30,000 | ~300,000 | 3 pairings |
| zkSync (batch) | PLONK | millions | ~400,000 | Amortized over batch |
| Semaphore v4 | UltraHonk | ~16,384 | ~250,000 | Larger circuit! |
| Worldcoin | Groth16 | ~600,000 | ~300,000 | Groth16 is cheapest |
| Aztec Connect | UltraPlonk | ~65,536 | ~400,000 | 10× larger circuit |

*\* Estimated, awaiting integration test measurement*

**Takeaway:** Credit Rail's circuit is remarkably compact at 6,248 gates — smaller than Tornado Cash (30K) and Semaphore (16K). Verification gas is dominated by the UltraHonk proof structure (pairing-based checks), not circuit size. UltraHonk is ~30% more expensive than Groth16 for verification but avoids the trusted setup ceremony entirely.

### Lending Protocol Gas (Non-ZK)

| Protocol | Borrow Gas | Notes |
|----------|------------|-------|
| Aave v3 `borrow()` | ~300,000 | No ZK, oracle reads |
| Compound v3 `withdraw()` | ~200,000 | No ZK |
| **Credit Rail `createLoan()`** | ~600,000* | ZK verification adds ~300k |
| **Credit Rail `activateLoan()`** | ~356,000 | Comparable to Aave borrow |

The ZK overhead (~300k gas) is the cost of **privacy-preserving underwriting**. This is a one-time cost per loan that enables the borrower to prove creditworthiness without revealing financials on-chain.

## 9. Optimization Opportunities

### High Impact (>50k gas savings)

| Optimization | Savings | Effort | Notes |
|-------------|---------|--------|-------|
| **Poseidon2 precompile** (if available on zkSync) | ~50,000 | Low | Replace library with native precompile |
| **Batch proof verification** (future) | ~200,000 per additional loan | High | Aggregate multiple proofs into one verification |
| **Groth16 migration** | ~100,000 | High | Cheapest verification but requires trusted setup |
| **Reduce Loan struct fields** | ~50,000 | Medium | Pack 18 fields into fewer storage slots |

### Medium Impact (10k–50k gas savings)

| Optimization | Savings | Effort | Notes |
|-------------|---------|--------|-------|
| Storage packing (Loan struct) | ~20,000 | Medium | Pack `tierId`, `state`, `aprBps` into one slot |
| Fewer CreditPolicy external calls in `createLoan()` | ~10,000 | Low | Cache reads or batch into single call |
| Remove redundant u64 checks (already in circuit) | ~5,000 | Low | Circuit already validates; on-chain is defense-in-depth |

### Low Priority

| Optimization | Savings | Notes |
|-------------|---------|-------|
| Assembly for keccak256 underwriter check | ~200 | Marginal |
| Inline `Field.toField()` | ~1,000 | Minor overhead |

### Not Recommended

| "Optimization" | Why Not |
|----------------|---------|
| Remove u64 range guards | Defense-in-depth against circuit bugs |
| Remove on-chain industry check | Not in circuit — must stay on-chain |
| Remove proof freshness check | Critical anti-replay defense |
